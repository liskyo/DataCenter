# ⚡ 智慧電力與任務感知調度系統 (Dynamic Power Capping & Carbon Tracking)

本模組為數據中心智慧監控平台（DCIM）的核心調度與防禦系統。特別針對電網容量吃緊、高價值 AI 算力高耗能的物理特性而設計，能在電網容量受限（如跳電風險、契約容量突降）時，透過「智慧閉環控制」自動下壓非核心伺服器能耗，實現「電力限額、算力不停擺」的動態調度，並同步實施秒級排碳積分追蹤。

---

## 🛠️ 一、 核心控制邏輯與演算法 (Closed-Loop Capping Engine)

系統基於**閉環反饋控制 (Closed-Loop Control)** 模型，每秒輪詢總用電量，並與電網的安全邊際閾值進行實時比對：

```
                    ┌────────────────────────┐
                    │ 遙測採集 (Telemetry)   │◀─────── 3 大 In-Band Telemetry Agents
                    └───────────┬────────────┘
                                │ (1s 輪詢)
                                ▼
                    ┌────────────────────────┐
   超出閾值? ───────▶│  閉環控制與超額調度    │
   (Deficit > 0)    └───────────┬────────────┘
                                │ (依 Tier 優先權自動降頻)
                                ▼
                    ┌────────────────────────┐
                    │ 物理硬體保護與算力折損 │───────► 動態寫入 power_kw & FLOPS 折損
                    └────────────────────────┘
```

### 1. 任務與設備優先級定義 (Priority Tagging)
為保障高價值的 AI LLM 訓練任務，系統將伺服器劃分為三個層級（Tier）：
*   **Tier 1 (Critical - 核心 AI 算力)**：核心 GPU AI 伺服器（如 GB200、H100/H200 機櫃），執行核心大型語言模型訓練。此類伺服器在限電時**擁有最高保護級別，絕不主動降頻**。
*   **Tier 2 (Normal - 一般業務)**：通用計算或非核心 GPU 伺服器，負責日常推論或大數據處理。當超載嚴重時，才會適度實施降頻。
*   **Tier 3 (Background - 背景任務)**：通用 CPU 通用伺服器，執行備份、日誌分析等背景任務。**最先被犧牲，能在毫秒內下壓至最低能耗底線**。

### 2. 高端硬體「能耗物理底線」安全保護
在自動限電時，系統絕不盲目下壓功率以致伺服器死機，而是嚴格遵守硬件物理極限：
*   **高端 AI GPU 伺服器 (8x GPU)**：預設最大功耗為 `8.0 kW`，硬體能耗極限底線為 **`3.0 kW`**。
*   **通用 CPU 伺服器**：預設最大功耗為 `1.5 kW`，硬體能耗極限底線為 **`0.6 kW`**。

## 📡 二、 遙測管線與高強健性設計

系統之底層通訊與同步邏輯在實踐中進行了極致的強健度加固（Robustness Engineering），以應對 Windows 環境與高併發下常見的邊界 Bug：

### 1. 併發同步防禦：重入鎖 (Reentrant Lock) 升級
*   **痛點**：底層數據在多執行緒遙測上報與 API 調度時需要並行防護。傳統 `threading.Lock()` 在「鎖上加鎖」的嵌套調用下會引發**自我死鎖 (Deadlock)**。
*   **解決方案**：升級為 **`threading.RLock()`**（重入鎖），允許同一個執行緒在持有鎖的情況下，安全地重複進入子方法，徹底根除並行死鎖隱患。

### 2. Windows 控制台 CP950 編碼異常兜底
*   **痛點**：科學告警文字中含有 `≤` 等科學符號，在 Windows 繁中控制台（預設 CP950 編碼）列印時會引發 `UnicodeEncodeError`，進而引發 API 500 內部伺服器崩潰。
*   **解決方案**：在日誌列印中實施雙重 `try-except` 保護，並在編碼衝突時將 `≤` 自動轉換為標準 ASCII 的 `<=` 進行降級輸出，**確保遙測管線永不中斷**。

### 3. Windows IPv6/IPv4 雙棧解析防禦
*   **解決方案**：前端 API 基礎連線由 `localhost` 全面替換為明確的 IPv4 實體環回位址 **`127.0.0.1`**，徹底避開 Windows 系統中 `localhost` 優先解析為 IPv6 `[::1]` 所導致的 `ERR_CONNECTION_REFUSED` 連線拒絕巨坑。

## 🎭 三、 COMPUTEX 展場現場互動演示優化

為了在展場中為貴賓提供最富衝擊力的視覺效果，系統內建了專屬的 **「電網緊急限電模擬器」**：

### 1. 一鍵模擬容量暴跌至 `50 kW`
*   當點擊「模擬電網容量暴跌 50%」時，系統會將契約用電容量瞬間砍至 **`50.0 kW`**（95% 安全警戒線為 **`47.5 kW`**）。
*   由於當前機房的總負載（IT 設備約 `18.6 kW` + 廠務與冷卻固定能耗 `35.0 kW`）約為 **`53.5 kW`**，會**真實且立即地超出 `47.5 kW` 安全警戒水位**！

### 2. 斷崖式電力降頻自癒 (Closed-Loop Capping in Action)
*   一旦超限，閉環控制演算法會瞬間被觸發，將所有 **Tier 3 (Background)** 通用 CPU 伺服器的功率**強制壓死在 0.6 kW 的物理最低能耗底線**。
*   前端時序圖表中的 IT 功耗曲線會出現一個**震撼的斷崖式下滑**，並最終將總用電強行鎖定在 `47.5 kW` 以下，確保系統綠色安全運行！
*   **Tier 1 AI 伺服器** 依然維持 `8.0 kW` 的全功率滿載輸出，保障核心大模型訓練絕不斷電！

### 3. Redfish BMC 批次原則手動覆寫
*   管理員可以在右側伺服器陣列中多選多台設備，拉動右下方的 **「Redfish 批次電力原則控制面板」**，實時為這些設備手動分配優先級或限制功率上限。
*   介面會動態播放高質感的 **BMC Redfish 限制原則派發進度條**，展示與實體服務器硬體帶外管理的完美對接！
