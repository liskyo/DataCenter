# DataCenter 前端 3D 室內建模與互動體驗優化紀錄

**日期:** 2026-03-30
**目標專案:** AQUA·CTRL / DataCenter 3D 數位孿生監控系統

本文件記錄了今日針對 DataCenter 專案前端在 3D 機房渲染、UI 互動與操作體驗上的深度優化與修正項目。這些改進大幅提升了系統的視覺專業度 (Cyberpunk Style) 以及操作的直覺性與防呆機制。

---

## 一、 主要優化項目

### 1. 左側編輯工具列 (Sidebar Icons) 色彩重構 🎨
*   **問題**: 原本 `twins` 編輯模式下的設備新增按鈕皆為統一的冷灰色 (`slate-400`)，視覺層次不明確，在複雜的操作中缺乏辨識度。
*   **優化**: 賦予每種設備圖示獨特且具備霓虹科技感 (Cyberpunk/Neon) 的專屬色彩，並保留滑鼠懸停 (Hover) 的發光回饋。
    *   `Server`: 靛藍色 (`indigo-400`)
    *   `CRAC`: 冰湖綠 (`teal-400`)
    *   `PDU`: 亮橘色 (`orange-400`)
    *   `CDU`: 青藍色 (`cyan-300`)
    *   `UPS`: 備用能源黃 (`yellow-500`)
    *   `Chiller`: 科技水冷藍 (`blue-500`)
    *   `Dashboard`: 戰情綠 (`emerald-500`)
    *   `Network Rack`: 賽博紫 (`purple-500`)

### 2. 機房幾何空間動態生成 (Dynamic Room Context) 🏗️
*   **功能**: 機房從原本固定尺碼的寫死 (Hardcoded) 牆壁，升級為**可完全自訂長、寬、以及門口方位**的參數化 3D 場景。
*   **實作細節**:
    *   在 Zustand Data Store (`useDcimStore.ts`) 中的 `LocationData` 擴充了 `width`、`depth` 以及 `doorPosition` 資料結構。
    *   全面重寫 `RoomContext.tsx`，加入 `<Walls>` 渲染引擎。牆壁會根據設定好的長寬自動拼算座標，並在使用者指定的那面牆 (左/右/前/後) 動態挖空並安裝玻璃門禁讀卡通道。
    *   **視覺透視優化**: 預設隱藏面向視角的正面牆面 (`+Z`)，營造出高級的等距剖視圖 (Isometric Cutaway) 感，不再遮擋視線。

### 3. 機房屬性面板與「防爆防呆」刪除機制 🛡️
*   **功能**: 點擊空白地面時，右側屬性面板會呼叫出全新的 **Room Settings** 控制項，允許使用者即時調整名稱與房間尺寸。
*   **安全性**: 面板最下方新增了紅色的 **Delete Room** 按鈕。為了防止辛辛苦苦擺設的機房被一鍵誤刪，加入了兩道防線：
    1.  **數量檢核**: 系統內必須至少保留一個機房場景。
    2.  **密碼確認**: 強制要求輸入系統管理員密碼 (`admin`) 才能通過並真正執行刪除指令。

### 4. 移除 3D 的傾倒干擾，導入「按鈕化轉向」 ⚙️
*   **問題**: `@react-three/drei` 內建的 `PivotControls` 拖曳介面會同時顯示 X、Y、Z 軸的三種旋轉環。但在機房中，伺服器機櫃與設備只需要水平轉向 (Yaw)，不需要前後左右傾倒 (Pitch/Roll)，不小心拉到通常只會造成視覺災難。
*   **優化**:
    *   全面將 3D 模型控制器的 `disableRotations` 設為 `true`，讓 3D 畫面上只留下乾淨的「平移十字箭頭」。
    *   在右側的機櫃/設備屬性面板新增 **Direction Control** 區塊，提供最直覺的 `[ ↺ 左轉 90° ]` 與 `[ ↻ 右轉 90° ]` 大按鈕。完美解決視角微調與手震的問題。

### 5. 高度擬真的通訊線路 (Cabling Routing) 🔌
*   **問題**: 先前的網管線路視覺只是簡單地從「A 機櫃」拉一條線到「B 機櫃頂部」，缺乏細緻度。
*   **優化**:
    *   修改 `NetworkLines.tsx`，改以「**伺服器為單位**」進行迴圈運算。
    *   現在**每一台伺服器**都會獨立拉出一條網線，並且在空間內部動態計算該伺服器的 `U` 高度座標（包含機櫃自身的轉向角度）。
    *   線材的終點會精準對接到 Network 機櫃內的 **Core Switch 設備背板高度**。若該網路機櫃尚未安插 Switch，系統會透過 `useDcimStore.ts` 賦予其預設的連線熱點，創造極具寫實感的多線流排線視覺。

## 二、 核心 Bug 修復 (Bug Fixes)

### 🚨 修正 U42 頂層伺服器被機櫃屋頂吞沒的問題
*   **情境**: 當使用者在機櫃最上方 (U42) 排入設備時，設備無法在 3D 畫面中正確渲染出來。
*   **根源**: 原本的機櫃總高度 (`rackHeight`) 運算邏輯只有加上 `0.1m` 的緩衝空間。但在整體渲染中，機櫃同時擁有底部基座 (`0.1m`) 以及頂部屋頂 (`0.1m`)。這導致伺服器整體向 Y 軸墊高後，最上層的 U42 伺服器剛好「完美卡進」了屋頂的幾何方塊內部，被其材質擋住。
*   **解法**: 修改 `RackModel.tsx`：
    ```typescript
    // Before:
    // const rackHeight = data.uCapacity * U_HEIGHT + 0.1;
    // After: 增加到 0.2m (底座 0.1m + 屋頂 0.1m)
    const rackHeight = data.uCapacity * U_HEIGHT + 0.2;
    ```
    修復後，不論是網格邊框還是實體屋頂，都能順利貼合在 U42 設備的上方，不會再發生模型交錯重疊 (Clipping) 遮擋的狀況。

## 三、 未來開發建議
1.  **效能監控**: 隨著伺服器排線變成 M × N 的多線條渲染（尤其是整排機櫃滿載 42U 伺服器時），`QuadraticBezierLine` 的數量可能引發 Three.js Draw Calls 暴增，後期可考慮導入 InstancedMesh 或 LineSegments 進行合併優化。
2.  **機房長寬極限**: 目前 UI 輸入限制為 `5m ~ 100m`。若房間大小超過 50m，預設的 `<pointLight>` 與 `<ContactShadows>` 的範圍涵蓋可能不足，未來可寫成動態擴展光源方案。
