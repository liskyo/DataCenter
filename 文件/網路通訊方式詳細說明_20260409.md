# DCIM 網路通訊方式詳細說明

本文件說明目前 `Network` 頁面可設定的設備通訊方式，提供維運、整合與現場部署時參考。  
適用設備類型：`Server`、`Rack`、`Tank`、`CDU`、`Switch`。

---

## DCIM 與資料中心通訊協定整理（總覽）

### 1) 概述

DCIM（Data Center Infrastructure Management）用於整合與監控資料中心內的：

- IT 設備（Server、Switch）
- 設施設備（CDU、UPS、PDU、HVAC）
- 環境感測器（溫度、濕度、漏水）
- 電力與冷卻系統

核心能力在於：  
**透過多種通訊協定收集 telemetry 資料，並進行統一管理與可視化。**

---

### 2) DCIM 支援的主要通訊協定

#### 2.1 IT 設備類

##### SNMP
- 用途：傳統網路設備與部分 server / PDU
- 模式：Polling + Trap
- 特性：
  - 廣泛支援
  - 標準成熟
  - 即時性較低

##### Redfish
- DMTF 推動的 RESTful 硬體管理標準
- 常用於 server BMC
- 特性：
  - JSON over HTTP
  - 易於自動化
  - 支援事件與 telemetry

##### IPMI（逐漸淘汰）
- 傳統 BMC 管理協定
- Binary protocol
- 安全性與可用性通常不如 Redfish

---

#### 2.2 網路設備類

##### Streaming Telemetry（OpenConfig + gNMI）
- OpenConfig 提供資料模型，gNMI 提供傳輸介面（gRPC）
- 特性：
  - Push-based（非 polling）
  - 高頻率資料收集
  - 支援訂閱（subscribe）
  - 適用於大規模網路

##### NETCONF / RESTCONF
- 用於設備配置與部分 telemetry
- 基於 YANG data model

##### SNMP
- 仍廣泛用於 legacy network devices

---

#### 2.3 設施 / OT 類

##### Modbus
- 工業控制協定
- 常見於 UPS、發電機、電力設備

##### BACnet
- 建築自動化協定
- 常見於 HVAC、冷卻、環境控制

##### MQTT
- Pub/Sub 架構
- 適用 IoT 與分散式感測器

---

### 3) 各類設備支援的通訊方式

| 設備 | 常見協定 | 資料傳輸方式 |
|------|----------|--------------|
| Server / BMC | Redfish / IPMI / SNMP | Pull / Push / Event / Streaming |
| Switch | gNMI / NETCONF / SNMP | Pull / Streaming / Event |
| CDU | Modbus / BACnet / Redfish | Pull / Event / Streaming（部分） |
| Rack / PDU | SNMP / Modbus / MQTT | Pull / Event / Streaming（間接） |

## 現行系統運作架構（As-Is）

目前 DCIM 系統的通訊與資料處理路徑可摘要如下：

1. 設備或 Agent 產生遙測資料（**設備端**）  
2. 資料進入後端入口（主要為 `POST /ingest`，**DCIM 後端**）  
3. 後端將資料寫入 Kafka Topic（預設 `telemetry`，**DCIM 後端**）  
4. Kafka Consumer Worker 持續訂閱並取出資料（**DCIM 後端**）  
5. 應用層進行資料清洗、ID 對應、告警判斷與異常偵測（**DCIM 後端**）  
6. 寫入時序資料（InfluxDB）與告警儲存（MongoDB，**DCIM 後端**）  
7. 透過 SSE 廣播給前端頁面，即時更新監控畫面（發送端：**DCIM 後端**；接收端：**DCIM 前端**）

簡化架構圖：

```text
[設備端] Server/Rack/Tank/CDU/Switch
          |
          v
   [設備端] Agent / Device Push
          |
          v
 [DCIM 後端] FastAPI: POST /ingest
          |
          v
 [DCIM 後端] Kafka Topic: telemetry
          |
          v
 [DCIM 後端] Kafka Consumer Worker
          |
          v
[DCIM 後端] Telemetry/Alert Processing
    |                 |
    v                 v
[DCIM 後端] InfluxDB   [DCIM 後端] MongoDB
    \               /
     \             /
      v           v
      [DCIM 後端] SSE Broadcast
              |
              v
      [DCIM 前端] Frontend Dashboards
```

### 目前架構重點

- 既有系統以 Kafka 作為核心事件匯流排，前後端即時資料同步以此為主幹。
- `Network` 頁面目前是「設定與說明層」，預設不直接改動後端執行流程。
- 通訊方式選項調整採「不影響現行運作」原則，先控管前端可配置範圍。
- 層級切分可簡化為：**設備端（資料來源）→ DCIM 後端（接收/處理/儲存/廣播）→ DCIM 前端（展示/互動）**。

## 現行資料主幹詳細說明（/ingest → Kafka → Consumer → Processing → DB/SSE）

以下針對你指定的主幹路徑做逐段說明：

### A. `[DCIM 後端] FastAPI: POST /ingest`

**角色**
- 系統的統一資料入口，接收設備端/Agent 送來的遙測資料。

**主要責任**
- 接收 JSON payload（例如 `server_id`、`temperature`、`cpu_usage`、`timestamp`）。
- 進行基本接收與交棒，不在入口層做重運算。
- 將資料送往 Kafka（`telemetry` topic）以進入非同步處理主幹。

**輸入 / 輸出**
- 輸入：設備端上送的 HTTP POST。
- 輸出：資料事件進入 Kafka；API 回應成功/失敗狀態給來源端。

**失效影響**
- 若 `/ingest` 無法服務，設備端上送會失敗或超時，資料無法進入後續流程。

**維運重點**
- 保持入口快速，避免同步阻塞。
- 失敗時應有明確錯誤碼與可追蹤 log。

---

### B. `[DCIM 後端] Kafka Topic: telemetry`

**角色**
- 事件匯流排與緩衝層，解耦「接收」與「處理」。

**主要責任**
- 暫存與傳遞遙測事件，承接尖峰流量。
- 讓多個下游消費者可獨立擴充，不互相耦合。

**輸入 / 輸出**
- 輸入：`/ingest` 或其他 producer 寫入事件。
- 輸出：由 consumer worker 拉取後進入業務邏輯。

**失效影響**
- 若 broker 不可用，入口雖可收到請求但事件無法可靠交付至後續處理。

**維運重點**
- 監控 broker 可用性、topic 堆積、consumer lag。
- 監控 group rebalance 頻率，避免頻繁 Join/Leave。

---

### C. `[DCIM 後端] Kafka Consumer Worker`

**角色**
- 持續從 `telemetry` 消費事件，驅動後端資料處理流程。

**主要責任**
- 使用 poll 週期取回訊息並逐筆交給處理層。
- 在異常時重連並維持 consumer 存活。

**輸入 / 輸出**
- 輸入：Kafka `telemetry` 訊息。
- 輸出：傳入 `Telemetry/Alert Processing` 進行邏輯判斷與落庫。

**失效影響**
- 入口仍可收資料，但無人消費時會造成 topic 累積，前端即時資料延遲。

**維運重點**
- 檢查 consumer 是否長駐穩定、是否頻繁重啟。
- 監控每秒處理量與錯誤率。

---

### D. `[DCIM 後端] Telemetry/Alert Processing`

**角色**
- 資料語意化核心：把原始遙測轉成可監控、可告警、可追溯的結果。

**主要責任**
- ID 正規化與資產對應（asset/display mapping）。
- 模擬/真實資料衝突保護與電源狀態注入。
- 閾值判斷（溫度 warning/critical）、異常偵測（AI anomaly）。
- 液冷指標檢查（流量、壓力、漏液、液位等）。

**輸入 / 輸出**
- 輸入：consumer 取回的 telemetry payload。
- 輸出：
  - 時序數據寫入 InfluxDB
  - 告警事件寫入 MongoDB
  - 最新狀態透過 SSE 廣播前端

**失效影響**
- 即便資料進入 Kafka，也可能無法形成告警或可視化結果。

**維運重點**
- 閾值要與前端顯示一致，避免「後端判斷」與「前端顏色」不一致。
- 新增設備型別時要同步更新處理邏輯與欄位驗證。

---

### E. `[DCIM 後端] InfluxDB` 與 `[DCIM 後端] MongoDB`

#### InfluxDB（時序庫）
- **用途**：存連續指標（CPU、溫度、流量等）做趨勢與時間區間分析。
- **特性**：適合高頻寫入與時間序列查詢。

#### MongoDB（事件庫）
- **用途**：存告警與事件紀錄（告警類型、訊息、時間、設備）。
- **特性**：適合事件檢索、稽核追蹤與通知對帳。

**雙庫分工原因**
- 時序資料與事件資料查詢模式不同，分庫可提高效能與可維運性。

---

### F. `[DCIM 後端] SSE Broadcast`（對 `[DCIM 前端]`）

**角色**
- 將後端最新狀態即時推送到前端監控畫面。

**主要責任**
- 當處理層完成每筆資料判斷後，立刻廣播給已連線頁面。
- 降低前端輪詢頻率需求，提升即時感。

**輸入 / 輸出**
- 輸入：處理完成後的最新 payload。
- 輸出：前端儀表板與監控頁面即時更新。

**失效影響**
- 資料仍可能落庫，但前端即時畫面會延遲或停滯。

**維運重點**
- 監控連線數、斷線重連、廣播延遲。
- 確保前端有重連策略與降級顯示。

---

### 端到端總結

- **入口層**：`POST /ingest` 負責可靠接收。
- **傳輸層**：Kafka `telemetry` 負責解耦與緩衝。
- **處理層**：Consumer + Processing 負責判斷、告警與資料語意化。
- **儲存層**：InfluxDB（時序）+ MongoDB（事件）分工。
- **呈現層**：SSE 對前端做即時更新。

這條主幹的目標是：**即時、穩定、可追蹤、可擴充**。

## 1) Agent 主動 pull（現行預設）

### 說明
- 由設備側 Agent 主動依固定節拍收集資料後，送入 DCIM 既有資料流程。
- 在目前系統中，通常搭配 `/ingest` 或 Kafka 匯入路徑使用。

### 適用場景
- 已部署自有 Agent 的伺服器或設備。
- 需要快速整合、可自定義欄位與上報節奏。

### 優點
- 導入快、彈性高、可擴充欄位。
- 可配合設備現況做客製資料前處理。

### 注意事項
- 需管理 Agent 版本與部署生命週期。
- 需注意 API 可用性與重試退避策略。

---

## 2) Redfish 輪詢（Redfish Poll）

### 說明
- DCIM 以週期性輪詢方式向設備 BMC（Baseboard Management Controller）讀取狀態資料。
- 常見於標準化伺服器硬體監控（溫度、功耗、風扇、健康狀態等）。

### 適用場景
- 支援 Redfish 的伺服器或機櫃設備。
- 偏好標準協定、降低自研 Agent 依賴。

### 優點
- 協定標準化，跨廠牌可維持一致整合方式。
- 管理面向明確，利於資產盤點與維護。

### 注意事項
- 輪詢頻率過高會增加設備管理介面負載。
- 需妥善處理認證資訊與憑證安全。

---

## 3) 主動式訂閱（Kafka / SSE）

### 說明
- 設備或中介服務以事件流方式主動推送資料，DCIM 以訂閱方式接收。
- 目前系統設計中以 Kafka 為主要事件匯流排，前端即時視圖可透過 SSE 更新。

### 適用場景
- 即時性要求高（告警、異常、瞬時負載變化）。
- 多系統共用同一串流資料來源。

### 優點
- 延遲低、吞吐高、擴充性佳。
- 易於與即時分析、告警與後續資料管線整合。

### 注意事項
- 需確保 Broker 穩定性與 consumer group 配置正確。
- 需監控 Rebalance、Lag、連線中斷與重連行為。

---

## 4) SNMP 輪詢 + Trap

### 說明
- 以 SNMP Poll 週期拉取設備指標，並搭配 Trap 接收設備主動事件通知。
- 常用於交換器、網通設備與部分機房基礎設施。

### 適用場景
- 傳統網通設備監控（介面流量、埠狀態、硬體告警）。
- 現場設備普遍提供 SNMP 管理能力。

### 優點
- 相容性高、部署成熟、網通場域普遍可用。
- Poll + Trap 可兼顧穩定監測與事件即時通報。

### 注意事項
- 社群字串、ACL、版本（v2c/v3）需嚴格管理。
- OID 對照與 Trap 解碼需有標準維運流程。

## 暫不提供為選項（依目前頁面需求）

- `modbus_poll`
- `http_ingest_push`
- `kafka_producer`

> 註：上述項目可在後續版本按需求重新開放，不影響現行後端資料流程。

---

## 設定建議（實務）

- `Server`：優先 `Agent 主動 pull` 或 `Redfish 輪詢`
- `Rack`：優先 `主動式訂閱`
- `Tank`：若現場已有事件流，建議 `主動式訂閱`
- `CDU`：優先 `主動式訂閱`
- `Switch`：優先 `SNMP 輪詢 + Trap`

---

## 變更原則

- 本頁面設定以「不影響現行運作」為優先。
- 若需將頁面設定正式寫入後端（跨瀏覽器共享），建議串接後端 `integration profile` API 再上線。
