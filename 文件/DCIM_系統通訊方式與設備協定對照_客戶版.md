# DCIM 系統通訊方式與設備協定對照（客戶版）

> 用途：對客戶說明 DCIM 如何串接資料中心設備、各協定適用情境與商業價值。

## 一、DCIM 為何採多通道通訊

DCIM 若只靠單一協定，容易出現監控盲區。  
本方案採 **4 種通訊方式互補**：

1. **Agent 主動 Pull**
2. **Redfish 輪詢**
3. **Streaming Telemetry（gNMI / OpenConfig）**
4. **BMC 事件推送（Event Push / Webhook）**
5. **SNMP 輪詢與告警（Polling + Trap/Inform）**

可同時滿足：
- **即時性**：事件推送 + 串流遙測
- **完整性**：Agent + 輪詢補齊資料面
- **可靠性**：單一路徑中斷仍有其他通道可補位

---

## 二、四種通訊方式（客戶說明版）

### 1) Agent 主動 Pull

**說明**  
在主機或設備側部署輕量 Agent，主動蒐集 OS 與應用層指標，再送回 DCIM。

**常見資料**
- CPU、Memory、Disk、Process
- 應用服務健康、主機自訂 KPI

**優點**
- 對舊設備與異質環境相容性高
- 可依客戶需求客製採集欄位

---

### 2) Redfish 輪詢

**說明**  
透過 Redfish API 週期輪詢 BMC，取得標準化硬體與健康狀態。

**常見資料**
- 溫度、風扇、電源供應器、機箱健康
- 資產資訊（CPU/DIMM/NIC/FRU）
- 電源狀態與硬體告警

**優點**
- 跨品牌一致性高、導入快
- 不需進入 OS，即可監控硬體層

---

### 3) Streaming Telemetry（gNMI / OpenConfig）

**說明**  
設備持續推送高頻遙測資料至 DCIM，而非傳統高頻輪詢。

**常見資料**
- 介面流量、錯誤率、丟包、延遲
- 交換器 CPU/記憶體、Queue、Buffer

**優點**
- 低延遲、近即時
- 高頻資料傳輸效率高，適合網路核心監控

---

### 4) BMC 事件推送（Event Push）

**說明**  
設備發生異常時主動推送事件到 DCIM，不必等待輪詢週期。

**常見資料**
- 過熱、風扇故障、PSU 故障、硬體錯誤
- 開關機/重啟等管理事件

**優點**
- 告警最快速（事件驅動）
- 易於串接通知與工單流程

---

### 5) SNMP（Simple Network Management Protocol）

**說明**  
透過 SNMP 進行設備狀態輪詢（Polling），並接收設備主動送出的告警（Trap/Inform）。

**常見版本**
- **SNMPv2c**：部署簡單、相容性高
- **SNMPv3**：支援帳密驗證與加密，企業環境建議優先

**常見資料**
- 介面流量、錯誤封包、丟包、Port up/down
- CPU、記憶體、溫度、電源、風扇
- UPS/PDU 電力參數（電壓、電流、負載、電池）

**優點**
- 老舊設備與多品牌網通設備支援度最高
- 導入成本低，適合既有環境快速納管

## 三、設備 × 協定 × 常讀取資料（重點表）

| 設備類型 | 主要協定 | 常讀取資料 | 建議用途 |
|---|---|---|---|
| Server（伺服器） | Redfish + Agent + BMC Event | CPU/Memory、溫度、風扇、電源、硬體告警 | 同步掌握 OS 負載與硬體健康 |
| Switch（交換器） | gNMI/OpenConfig（主）+ SNMP（輔） | Port 流量、錯誤、丟包、CPU、光模組資訊、Trap 告警 | 高頻即時網路監控與異常定位 |
| CDU（液冷分配） | SNMP + Agent | 供回水溫、流量、壓力、液位、泵浦狀態、漏液告警 | 液冷安全監控與熱風險預警 |
| CRAC/CRAH（精密空調） | SNMP | 溫濕度、送回風、壓縮機/風機狀態、告警碼、設備運轉狀態 | 空調效能監控與節能優化 |
| UPS/PDU（電力） | SNMP | 電壓、電流、功率、負載率、電池健康、回路告警、Trap 異常通知 | 配電風險預警與容量管理 |
| Storage（儲存） | SNMP/Redfish(部分) | 容量、IO 延遲、控制器健康、磁碟狀態、硬體告警 | 避免儲存瓶頸與服務降級 |
| BMC（管理控制器） | Redfish + Event Push | Sensor、資產清單、事件日志、電源控制 | 硬體維運標準化與快速告警 |

## 三之一、SNMP 對客戶說明重點（可口播）

- **SNMP 在 DCIM 的角色**：作為跨品牌設備的「通用資料層」，特別適合網路、電力與既有機電設備。  
- **Polling + Trap 雙軌**：  
  - Polling 負責趨勢與定期盤點  
  - Trap/Inform 負責即時事件上報  
- **安全建議**：正式環境以 `SNMPv3` 為主，避免僅使用 v2c 社群字串。  
- **典型納管設備**：Switch、UPS、PDU、CRAC/CRAH、部分 Storage 與環控設備。  
- **商業價值**：在不更換既有設備前提下，快速提升可視化與告警能力。

## 四、對客戶可直接使用的話術（30 秒）

我們的 DCIM 不是依賴單一協定，而是用多通道整合 IT 與機電設備：  
IT 偏向 **Redfish / Agent / gNMI**，機電偏向 **SNMP**，再由 **事件推送** 做即時告警。  
這種架構可在既有設備上快速落地，同時達成 **即時、完整、可擴充、可持續運維**。

## 五、常見客戶問答（簡版）

### Q1：為何不用單一協定？
單一協定容易有資料盲區。多通道可同時兼顧即時與完整，且具備容錯能力。

### Q2：導入會不會很複雜？
可分階段導入：先 Redfish/BMC，再 Agent，最後加上 gNMI 高頻串流。

### Q3：會不會增加網路負擔？
不會。高頻資料走串流、低頻狀態走輪詢、告警走事件推送，整體更有效率。

### Q4：可否整合既有 NMS/ITSM/CMDB？
可以。透過 API 與事件介接，不需重做既有流程。

## 六、總結

DCIM 的核心不是「把資料收進來」而已，而是透過正確協定分工，做到：
- 更快告警（MTTD 下降）
- 更快定位修復（MTTR 下降）
- 更少停機與人工巡檢成本

這就是多通道 DCIM 架構的實際商業價值。
