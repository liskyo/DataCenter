# DCIM 系統通訊方式與設備協定對照（客戶版）

> 用途：用客戶可理解的方式說明 DCIM 如何整合 IT 與機電設備，並保留與本專案現況一致的描述。

---

## 一、為何採多通道整合

資料中心設備來源很多，單一協定通常無法同時覆蓋：

- 主機作業系統指標
- 硬體健康狀態
- 網通設備事件
- 電力與冷卻設備資料

因此 DCIM 會採用多通道整合，而不是只依賴一種協定。

---

## 二、常見整合方式

### 1) Agent / HTTP Push

- 適合伺服器 OS 指標與客製欄位
- 可收集 CPU、Memory、Process、應用 KPI
- 導入快，適合做即時遙測主幹

### 2) Redfish

- 適合伺服器 BMC 與硬體健康監控
- 可取得溫度、風扇、電源、資產資訊
- 也可延伸到電源控制與事件整合

### 3) Streaming Telemetry / gNMI

- 適合交換器與高頻網路遙測
- 能提供更低延遲與更高更新頻率

### 4) SNMP

- 適合既有網通、UPS、PDU、空調等設備
- 常搭配 Polling + Trap
- 導入成本低，適合快速納管舊設備

### 5) Event Push / Webhook

- 適合將硬體事件主動送到平台
- 有助於告警與工單自動化

---

## 三、設備與協定建議

| 設備類型 | 建議協定 | 主要價值 |
| --- | --- | --- |
| Server | Agent + Redfish | 同步掌握 OS 負載與硬體健康 |
| Switch | SNMP / gNMI | 監控流量、錯誤與埠狀態 |
| UPS / PDU | SNMP / Modbus | 電力風險與容量監控 |
| CDU / 冷卻設備 | API / SNMP / Modbus | 液冷與溫控安全 |
| BMC | Redfish + Event Push | 快速告警與控制延伸 |

---

## 四、與本專案現況的對齊說明

本專案目前已落地的主資料通道是：

```text
Agent / Simulation
   -> HTTP /ingest
   -> Kafka
   -> Backend processing
   -> InfluxDB / MongoDB
   -> SSE
   -> Frontend
```

也就是說：

- 目前最成熟的主幹是 Agent / HTTP Push
- Redfish、SNMP、gNMI 等屬重要整合方向
- 客戶溝通時可以說明多協定架構，但需區分已落地與規劃中能力

---

## 五、對客戶可直接使用的簡版話術

我們的 DCIM 採用多通道整合方式：  
主機層可用 Agent 收集即時遙測，硬體層可用 Redfish，網通與機電設備可用 SNMP 或其他標準協定。  
這樣可以在不推翻既有設備的前提下，同時做到即時、完整、可擴充的監控能力。

---

## 六、常見問答

### Q1：為何不用單一協定？

因為不同設備擅長的協定不同，多通道整合才能降低監控盲區。

### Q2：導入會不會很複雜？

可分階段導入，通常先從最容易接的 Agent 或標準協定開始。

### Q3：能否保留既有設備？

可以，多數既有設備可先用 SNMP 或現有 API 納管。
