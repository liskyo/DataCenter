# DataCenter 新人訓練可授課教師手冊（完整版）

## 文件用途與使用方式

本手冊提供講師可直接上課的完整教案，適用於 DataCenter 專案新人培訓。內容以「從 0 到可維運」為主軸，將專案拆成 8 章，涵蓋環境、架構、後端、前端、資料流、安全、除錯與實戰驗收。

- 適用對象：新進軟體工程師（前端、後端、全端）
- 課程形式：實作導向，講師示範 + 學員跟做 + 任務驗收
- 建議時數：總計 12~16 小時（可拆 2 天或 6~8 堂）
- 預期成果：學員可獨立啟動系統、追蹤一筆資料全鏈路、完成小功能開發與基本故障排查

---

## 一、課程總覽（講師版）

### 1.1 教學總目標

1. 讓學員理解本專案的核心價值與系統邊界。
2. 建立「資料流思維」：從資料進入到畫面呈現的完整生命週期。
3. 培養可實際交付的能力：能新增小功能、定位問題、說明設計理由。
4. 建立工程習慣：分層、日誌觀測、錯誤處理、安全意識。

### 1.2 教學方法與節奏

每章建議使用固定節奏，提升吸收效率：

- 10% 前測提問：確認學員基礎與盲點。
- 30% 講師示範：先看正確方法與思考路徑。
- 45% 學員實作：實際操作，講師巡迴指導。
- 15% 回顧收斂：釐清概念、指出常見錯誤、統一最佳實務。

### 1.3 教師課前檢查清單

- [ ] 學員電腦版本符合最低要求（Node、Python、Docker）。
- [ ] 所有學員可存取專案程式碼與必要憑證（僅測試憑證）。
- [ ] 講師機器可穩定啟動完整系統（前端、後端、資料服務）。
- [ ] 課程範例資料可重現（至少 3 種情境：正常、警告、告警）。
- [ ] 每章驗收題有標準答案與 demo 螢幕錄影備援。

### 1.4 教學產出與驗收機制

每章要求學員至少產出以下內容：

1. 一份操作紀錄（步驟與結果）。
2. 一份問題紀錄（遇到錯誤、排查、結論）。
3. 一次口頭解釋（用 2~3 分鐘說明本章核心流程）。

## 第 1 章：專案全貌與架構地圖（90 分鐘）

### 1. 教學目標

- 認識專案角色：前端、後端、資料層、中介層、運維層。
- 能口頭講出一筆遙測資料如何被接收、處理、儲存與顯示。
- 能辨識「主路徑」與「降級路徑」差異。

### 2. 講師授課流程（建議）

1. **破冰提問（10 分）**  
   請學員回答：「你認為資料中心監控系統最怕什麼？」  
   教師解答：最怕「看不到、看太慢、看不準」。對應為可觀測性、低延遲與資料可信度。  
   目的是讓學員從技術導向切到業務導向（可用性、即時性、可靠性）。

2. **架構圖導覽（25 分）**  
   講師畫出分層：  
   Device/Agent -> Backend API -> Kafka/Fallback -> Processing -> DB -> SSE -> Frontend Dashboard。

3. **目錄走讀（25 分）**  
   引導學員辨識：  
   - `backend/`：API 與服務層  
   - `frontend/`：畫面與狀態  
   - `layout/`：場域布局資料  
   - 根目錄腳本：一鍵啟停

4. **案例演示（20 分）**  
   輸入一筆遙測，對照 log 與畫面更新，建立「全鏈路視角」。

5. **回顧小測（10 分）**

### 3. 講師重點台詞（可直接使用）

- 「先學資料去哪裡，再學每個檔案做什麼。先有全圖，細節才不會迷路。」
- 「主路徑是效能最佳解，降級路徑是生存線。兩者都要能說清楚。」
- 「你今天不是學 API 而已，是學一個可維運系統如何保證服務不中斷。」

### 4. 學員練習

- 練習 A：畫出你理解的架構圖（不可看講義，5 分鐘）。
- 練習 B：用一句話說明每個服務的責任邊界。

### 5. 教師標準解答

- 架構圖至少要包含：資料入口、處理層、儲存層、推送層、展示層。
- 主路徑需明確指出 Kafka；降級路徑需指出 fallback 直接處理。
- 若學員只講技術名詞，講師需追問「那失敗時怎麼辦？」以補齊韌性概念。

### 6. 常見誤區與矯正方式

- 誤區：把前端當成資料來源。  
  矯正：前端是展示與交互，不是資料真實來源。
- 誤區：以為 Kafka 掛掉就全系統不可用。  
  矯正：說明 fallback 設計與代價（吞吐、解耦下降）。

## 第 2 章：環境建置與啟動流程（90 分鐘）

### 1. 教學目標

- 能完成本機環境建置並成功啟動。
- 能分辨「一鍵啟動」與「手動啟動」的適用情境。
- 遇到常見啟動失敗可自行定位問題。

### 2. 講師授課流程

1. **環境前檢（20 分）**  
   檢查 Node、Python、Docker 版本與 daemon 狀態。

2. **一鍵啟動示範（20 分）**  
   示範腳本做了什麼：建立 venv、安裝依賴、啟動 compose、啟動 app。

3. **手動啟動演練（30 分）**  
   分步啟動基礎服務、後端、前端，讓學員理解依賴順序。

4. **錯誤情境排查（20 分）**  
   內建故障題：埠被占用、Docker 未啟、環境變數錯、依賴缺失。

### 3. 課堂實作任務

- 任務 A：學員必須完成一鍵啟動。
- 任務 B：學員關閉服務後改用手動模式重新啟動。
- 任務 C：學員提交一份「啟動失敗排查清單」。

### 4. 教師標準解答與講評

- 一鍵啟動成功標準：前端可開啟、後端 API 回應正常、資料服務 healthy。
- 手動啟動成功標準：可說明每一步為何要先後順序。
- 排查清單必含：環境版本 -> 容器狀態 -> 埠占用 -> `.env` -> 應用 log。

### 5. 講師提醒（務必強調）

- 「能啟動不等於懂系統；懂依賴順序才算真正上手。」
- 「任何錯誤先收斂範圍，不要同時改 5 個地方。」

## 第 3 章：後端主流程與服務分層（120 分鐘）

### 1. 教學目標

- 看懂 FastAPI 入口、路由分工與容器注入。
- 理解遙測處理主流程與錯誤處理策略。
- 能指出新增功能應放在哪一層。

### 2. 授課內容結構

1. **入口與生命週期（20 分）**  
   `main.py` 的啟動流程、路由掛載、全域依賴。

2. **分層與責任（30 分）**  
   Router（對外協議）/ Service（業務邏輯）/ Storage（持久化）角色。

3. **遙測流程導讀（45 分）**  
   `/ingest` 接收 -> 檢核/正規化 -> Kafka 或 fallback -> process_message -> 儲存/推送。

4. **設計討論（25 分）**  
   為何 Router 要薄？為何狀態不該散落在多處？
   教師解答：Router 應專注協議與參數驗證，業務邏輯集中在 Service 才能降低耦合、提升可測性。狀態若分散會增加同步成本，導致除錯困難與不一致風險。

### 3. 學員實作題

- 題目 A：新增一個簡單 health 子路由（含錯誤回應格式）。
- 題目 B：在不破壞既有流程下，增加一個遙測欄位預處理。
- 題目 C：寫出一段你對分層設計的理由（至少 100 字）。

### 4. 教師解答說明

- A 標準：回應含服務狀態、時間戳、必要依賴狀態。
- B 標準：前置處理應放在 service 層或標準化流程，不應直接塞在路由函式。
- C 評分：是否提到可測試性、可維護性、耦合度與變更成本。

### 5. 進階引導問題（給快班）

- 若要引入非同步工作佇列，現有架構最小侵入切點在哪？
  教師解答：優先從 Service 層切入，將耗時計算改為事件投遞；Router 介面盡量不變，降低對上層影響。
- 若未來支援多租戶，哪些欄位與授權模型要先調整？
  教師解答：先補 `tenant_id` 與資料隔離策略，再調整 RBAC 為租戶範圍授權，避免跨租戶資料外洩。

## 第 4 章：資料層與中介服務（120 分鐘）

### 1. 教學目標

- 清楚區分各資料元件用途與效能取捨。
- 理解中介服務故障時的降級策略。
- 建立「資料一致性 vs 即時性」的工程判斷能力。

### 2. 授課流程

1. **Kafka（25 分）**  
   介紹解耦、緩衝、吞吐與可追溯性。

2. **Redis（20 分）**  
   說明 SSE pub/sub 的低延遲優勢與注意事項。

3. **Influx vs Mongo（35 分）**  
   時序資料與文件資料的分工，並比較查詢情境。

4. **故障演練（40 分）**  
   模擬 Kafka 不可用、Redis 延遲，觀察系統行為與回退效果。

### 3. 講師示範腳本（操作重點）

- 步驟 1：送入固定頻率遙測資料。
- 步驟 2：暫停中介服務，觀察資料是否進 fallback。
- 步驟 3：恢復服務，確認系統是否回到主路徑。
- 步驟 4：核對資料是否出現在正確資料庫與前端畫面。

### 4. 教師標準解答

- 為何 metrics 不放 Mongo：時序聚合成本高、查詢效能不佳。
- 為何 alerts 放 Mongo：事件型資料結構彈性高，便於歷史追蹤與稽核。
- 回退策略評估：可用性上升，但觀測與告警要同步加強，避免「降級常態化」。

### 5. 評分規準

- 40% 能正確說明元件責任
- 30% 能完整描述故障時資料流
- 20% 能提出至少一項監控建議
- 10% 表達清晰度

## 第 5 章：前端即時流、狀態管理與儀表板（120 分鐘）

### 1. 教學目標

- 理解前端如何接收 SSE 並在斷線時 fallback。
- 看懂狀態管理在畫面更新中的作用。
- 能新增一個簡易儀表板卡片且保持效能穩定。

### 2. 授課流程

1. **SSE hook 導讀（35 分）**  
   連線建立、重連策略、心跳或逾時判斷、降級輪詢。

2. **首頁資料映射（35 分）**  
   API payload -> store -> UI cards/charts 的映射邏輯。

3. **狀態管理（25 分）**  
   哪些是 server state、哪些是 UI state，避免責任混淆。

4. **小改版實作（25 分）**  
   新增資料來源狀態標籤與告警摘要卡片。

### 3. 學員任務

- 任務 A：新增連線狀態徽章（SSE / Polling / Disconnected）。
- 任務 B：新增最近 N 筆告警摘要區塊。
- 任務 C：說明為何不把所有資料都存在全域 store。

### 4. 教師標準解答

- A：狀態來源應與連線邏輯一致，避免手動硬編碼。
- B：摘要卡應處理空資料與異常資料，避免畫面崩潰。
- C：過度全域化會提升耦合與重繪成本，應依資料使用範圍分層管理。

### 5. 常見錯誤與修正

- 錯誤：`useEffect` 依賴設計不當導致重複連線。  
  修正：穩定依賴與清理函式，避免每次 render 建立新連線。
- 錯誤：把 fallback 視為錯誤狀態。  
  修正：fallback 是韌性策略，不是失敗本身。

## 第 6 章：認證授權與安全治理（90 分鐘）

### 1. 教學目標

- 理解 JWT 驗證、RBAC 權限與 API Key 的邏輯。
- 建立最小權限與憑證管理觀念。
- 能評估一個 API 是否具備基本安全防護。

### 2. 授課流程

1. **驗證流程（25 分）**  
   token 生成、驗證、過期、錯誤回應。

2. **授權流程（25 分）**  
   角色到權限映射，路由保護策略。

3. **Agent API Key（15 分）**  
   機器到機器情境與風險管理。

4. **安全實務（25 分）**  
   `.env`、秘密管理、日誌脫敏、權限審核。

### 3. 課堂案例

- 案例 A：viewer 嘗試調用 control API，應回應何種錯誤？
  教師解答：應拒絕存取（通常 403），並回傳可稽核資訊（角色、資源、時間戳或追蹤 ID）。
- 案例 B：token 過期時前端應如何處理使用者體驗？
  教師解答：導向重新登入或 refresh 流程，保留必要上下文，避免重複請求造成風暴。
- 案例 C：若發現測試帳密被提交到 repo，應立即做哪些事？
  教師解答：先撤銷與輪替憑證，再清查存取紀錄與影響範圍，最後補上 secret scan 與流程防護。

### 4. 教師標準解答

- A：回應 403（或專案規範對應碼），並保留可稽核資訊。
- B：導向重新登入、保留必要狀態、避免重複請求風暴。
- C：立即撤銷憑證、輪替密鑰、清查存取紀錄、補強流程（pre-commit/CI 掃描）。

### 5. 評分觀察點

- 是否能清楚區分 authentication 與 authorization。
- 是否理解「前端不可見不等於後端安全」。
- 是否能提出可落地的憑證治理流程。

## 第 7 章：除錯、監控與維運 SOP（90 分鐘）

### 1. 教學目標

- 建立可重複使用的故障排查流程。
- 能快速判斷問題位於前端、後端、網路或資料服務。
- 能用標準格式回報 incident。

### 2. 標準排查 SOP（講師須要求背熟）

1. **確認現象**：具體錯誤訊息、影響範圍、重現步驟。
2. **分層定位**：UI -> API -> Service -> DB/Queue。
3. **檢視日誌**：抓同時間窗的關鍵 log。
4. **最小改動驗證**：一次只改一件事，保留對照。
5. **回報結論**：原因、處置、預防措施。

### 3. 課堂故障演練

- 題目 A：SSE 斷線但 API 正常。
- 題目 B：資料庫可用但畫面無更新。
- 題目 C：告警資料有寫入但前端摘要卡空白。

### 4. 教師標準解答

- A：優先檢查 SSE 端點、Redis pub/sub 與前端重連邏輯。
- B：檢查資料是否到達前端 store、欄位映射是否變更。
- C：多半為前端 mapping/filter 條件不匹配，或資料欄位名異動。

### 5. Incident 回報模板（給學員）

- 事件名稱
- 發生時間
- 影響範圍
- 重現步驟
- 觀察到的錯誤
- 初步判斷
- 臨時處置
- 根因
- 永久修正
- 後續追蹤項目

## 第 8 章：小專題實戰與最終驗收（120 分鐘）

### 1. 教學目標

- 整合前 7 章能力，完成小功能交付。
- 練習工程溝通：需求理解、設計取捨、驗收說明。

### 2. 專題題目（可擇一）

1. **後端題**：新增一條告警規則與對應 API 查詢條件。  
2. **前端題**：新增「設備健康分數」卡片與趨勢圖。  
3. **全端題**：新增維運按鈕（需權限控管 + 操作結果回饋）。

### 3. 驗收流程（講師主持）

1. 需求確認（10 分）
2. 設計說明（15 分）
3. 開發實作（60 分）
4. Demo 與問答（25 分）
5. 講師講評（10 分）

### 4. 驗收評分 Rubric

- 40% 功能正確性：符合需求、結果可重現。
- 25% 程式品質：命名、分層、可讀性、錯誤處理。
- 20% 系統影響：不破壞既有流程、具基本回歸驗證。
- 15% 溝通能力：能說明取捨與風險。

### 5. 教師標準解答參考（抽象版）

- 後端題：規則邏輯放 service、路由只做協議層、補必要測試點。
- 前端題：先定義資料來源與 fallback，再做視覺卡片，避免 UI 先行造成資料耦合。
- 全端題：先保護 API 權限，再做 UI 行為；回應需有成功/失敗可追蹤資訊。

## 四、常見問題（FAQ）與標準回答

### Q1：新人該先學前端還是後端？

建議先學「一筆資料全鏈路」，再依職能深入。先有全貌，局部學習才不會碎片化。
教師解答：先全鏈路再分工，可避免跨層問題無法定位，是新人最快形成系統感的方式。

### Q2：為何要同時教主路徑與降級路徑？

因為監控系統核心是可用性。只會主路徑，遇到故障無法維運；降級策略是實務必備能力。
教師解答：主路徑保效能，降級路徑保生存，兩者缺一就無法達成可維運目標。

### Q3：這份教案最重要的考核點是什麼？

不是背指令，而是能解釋「為何這樣設計」與「失敗時怎麼辦」。
教師解答：看的是工程判斷力與可重現交付能力，不是記憶力。

### Q4：如何判斷學員已可上手？

可獨立完成四件事即達標：啟動系統、追資料流、改一個小功能、寫出故障排查紀錄。
教師解答：四項都能獨立完成且可重現，即代表具備實務上手能力。

## 八、每章 90 分鐘逐分鐘講稿版（講師可直接照表授課）

## 第 1 章逐分鐘講稿：專案全貌與架構地圖（90 分鐘）

### 0-10 分鐘｜開場與目標對齊
- 講師說：今天不先看程式細節，先建立全貌，目標是每個人都能說出資料怎麼流動。
- 講師做：投影課程地圖（8 章）與本章成果標準。
- 學員做：寫下「你認為監控系統最重要的 1 個能力」。
- 檢核點：至少 80% 學員回答到可用性或即時性。

### 10-20 分鐘｜業務情境導入
- 講師說：資料中心場景下，延遲與中斷會造成什麼影響。
- 講師做：用 1 個故障故事（溫度異常未即時告警）串起系統價值。
- 學員做：2 人一組討論「若系統掛掉，第一時間誰會受影響」。
- 檢核點：學員能說出至少 2 個利害關係人（維運、值班、管理者）。

### 20-30 分鐘｜架構總圖第一輪
- 講師說：先看黑盒流程，不談實作細節。
- 講師做：白板畫 `Agent -> Backend -> Kafka/Fallback -> Processing -> DB -> SSE -> Frontend`。
- 學員做：抄下流程並標註「主路徑」與「降級路徑」。
- 檢核點：學員能指出 fallback 發生位置。

### 30-40 分鐘｜目錄與責任邊界
- 講師說：目錄是組織思維，不只是資料夾。
- 講師做：逐一導覽 `backend/`、`frontend/`、`layout/`、啟停腳本。
- 學員做：在筆記中寫每個目錄一句責任描述。
- 檢核點：能分清「展示層」與「處理層」。

### 40-50 分鐘｜資料流第二輪（加入失敗情境）
- 講師說：設計系統時要先問「失敗時怎麼活下來」。
- 講師做：補上 Kafka 失效時的 fallback 走法。
- 學員做：畫第二版流程圖（含錯誤分支）。
- 檢核點：流程圖含至少 1 個故障分支。

### 50-60 分鐘｜講師示範一筆資料生命週期
- 講師說：接下來看真實資料從進入到畫面更新。
- 講師做：送一筆遙測，對照 log 與前端畫面變化。
- 學員做：記錄 5 個事件節點（接收、處理、儲存、推送、展示）。
- 檢核點：學員能用自己的話重述順序。

### 60-70 分鐘｜學員白板練習
- 講師說：請你當講師，對同學講一次系統流程。
- 講師做：抽 2 組上台，各 3 分鐘。
- 學員做：互評（清晰度、正確性、完整度）。
- 檢核點：至少 1 組能完整講對主路徑與降級路徑。

### 70-80 分鐘｜誤區矯正
- 講師說：你們剛剛最常犯的錯，是把前端當核心處理點。
- 講師做：列出 3 個常見誤解並逐一修正。
- 學員做：把自己的流程圖改成最終版。
- 檢核點：流程圖責任邊界清楚，無層次混淆。

### 80-90 分鐘｜回顧測驗與作業
- 講師說：這章只考一件事，能否說清楚資料如何流。
- 講師做：5 題快問快答。
- 學員做：提交本章產出（流程圖 + 100 字摘要）。
- 檢核點：80% 以上學員可通過 4/5 題。

## 九、全提問對照教師解答庫（每一題皆可直接使用）

> 說明：本區把手冊內所有提問、快問快答、案例題、情境題統一整理，授課時可直接複製到投影片或講師備忘卡。

## 9.1 第 1 章提問與教師解答

### 提問 1
- 問題：你認為資料中心監控系統最怕什麼？
- 教師解答：最怕「看不到、看太慢、看不準」。對應三件事是可觀測性、低延遲、資料可信度。若只能看到部分資料，等同盲飛；若延遲太高，告警沒有時效；若資料品質差，決策會錯誤。

### 提問 2
- 問題：如果主路徑正常，為什麼還需要降級路徑？
- 教師解答：因為生產環境一定會遇到中介服務故障。降級路徑保的是服務持續可用，不是最佳效能。主路徑負責效率，降級路徑負責生存。

### 提問 3
- 問題：前端在整個系統中的主要責任是什麼？
- 教師解答：前端核心責任是展示、互動與狀態回饋，不是資料真實來源與最終決策中心。資料處理與持久化應在後端與資料層完成。

### 第 1 章快問快答建議題（對應 80-90 分鐘）
- 問題：請說出一筆資料從進來到顯示至少 5 個節點。  
  解答：接收（API）-> 分流（Kafka/fallback）-> 處理（規則/標準化）-> 儲存（Influx/Mongo）-> 推送（SSE）-> 前端顯示。
- 問題：哪個元件負責即時推送協作？  
  解答：Redis pub/sub 配合 SSE 管線。
- 問題：Kafka 掛掉會不會立即全系統停擺？  
  解答：不會，系統應走 fallback，但吞吐與解耦能力會下降。
- 問題：為什麼要先教全貌再教細節？  
  解答：先有架構心智圖，後續看檔案才不會碎片化。
- 問題：本章最關鍵能力是什麼？  
  解答：能清楚說明主路徑與降級路徑的資料流。

## 9.2 第 2 章提問與教師解答

### 提問 1
- 問題：一鍵啟動與手動啟動各自的價值是什麼？
- 教師解答：一鍵啟動重效率與一致性，適合快速 onboarding；手動啟動重理解與除錯，適合定位依賴問題。兩者都要會，才算真正可維運。

### 提問 2
- 問題：環境建置時應先檢查什麼？
- 教師解答：先檢查執行環境（Node、Python、Docker daemon），再檢查服務狀態（容器/埠），最後檢查配置（`.env`）。順序不可反。

### 提問 3
- 問題：遇到啟動失敗，你第一步會做什麼？
- 教師解答：先確認現象與範圍（哪個服務失敗），再查對應 log；不要同時改多處。先定位再修正，避免引入新變因。

## 9.3 第 3 章提問與教師解答

### 題目 A
- 問題：新增一個簡單 health 子路由（含錯誤回應格式）。
- 教師解答：回應至少包含 `status`、`timestamp`、`dependencies`。若依賴異常，回應應可辨識問題來源（例如 database/queue），並維持一致的錯誤格式以利觀測。

### 題目 B
- 問題：在不破壞既有流程下，增加一個遙測欄位預處理。
- 教師解答：預處理應放在 service 或標準化流程，不應直接塞進路由層。原則是 Router 薄、Service 厚，並保留向下相容（舊欄位仍可處理）。

### 題目 C
- 問題：寫出分層設計理由（至少 100 字）。
- 教師解答：標準答案需提到 4 點：可測試性、可維護性、低耦合、變更成本。加分點是能說明「需求改動時哪一層最小影響」。

## 9.4 第 4 章提問與教師解答

### 提問 1
- 問題：為什麼 metrics 建議放 Influx，不放 Mongo？
- 教師解答：metrics 是時序資料，常做時間區間聚合、趨勢查詢與降採樣。Influx 在時序場景效能更好；Mongo 可存但查詢成本高、維運成本也可能提高。

### 提問 2
- 問題：Redis 在本專案主要扮演什麼角色？
- 教師解答：主要是即時推送中介（pub/sub），協助事件快速分發，不是長期持久化主庫。

### 提問 3
- 問題：中介服務故障時，如何判斷是「降級可用」還是「不可用」？
- 教師解答：若核心資料仍可接收、處理、落地且前端有替代顯示（例如 polling），屬降級可用；若資料進不來或結果不可觀測，才是不可用。

### 第 4 章案例判斷題（對應 80-90 分鐘）
- 問題：Kafka 異常時資料仍能被接收但延遲上升，屬於哪一類？  
  解答：降級可用，需追蹤延遲與告警。
- 問題：Redis 異常導致即時畫面停更，但 API 查詢仍有資料，下一步？  
  解答：先啟用/確認 polling 退化機制，再排查 pub/sub 通道。
- 問題：資料寫入成功但趨勢圖空白，先查哪裡？  
  解答：先查前端 mapping 與查詢條件，再查資料欄位一致性。

## 9.5 第 5 章提問與教師解答

### 提問 1
- 問題：為什麼 fallback 不能視為錯誤狀態？
- 教師解答：fallback 是韌性設計的一部分，目標是「服務不中斷」。它表示系統降級，而非功能完全失敗。

### 提問 2
- 問題：為什麼不把所有 state 都放進全域 store？
- 教師解答：過度全域化會造成重繪放大、耦合升高與維護困難。應區分 server state 與局部 UI state，依作用範圍管理。

### 提問 3
- 問題：最容易造成前端即時功能不穩定的程式問題是什麼？
- 教師解答：常見是 `useEffect` 依賴設計錯誤，導致重複建立連線；其次是 cleanup 不完整，造成殘留 listener。

## 9.6 第 6 章提問與教師解答

### 案例 A
- 問題：viewer 嘗試調用 control API，應回應何種錯誤？
- 教師解答：應拒絕存取（通常 403），並回傳可稽核的錯誤資訊（角色、目標資源、時間戳或追蹤 ID）。

### 案例 B
- 問題：token 過期時前端應如何處理使用者體驗？
- 教師解答：引導重新登入，保留必要上下文，避免無限重試與請求風暴。若有 refresh 機制，需有明確失敗回退行為。

### 案例 C
- 問題：若發現測試帳密被提交到 repo，應立即做哪些事？
- 教師解答：立即撤銷與輪替憑證、盤點存取紀錄、評估影響範圍、補上掃描與流程防護（如 pre-commit/CI secret scan）。

### 第 6 章情境題（對應 80-90 分鐘）
- 問題：Authentication 與 Authorization 差異？  
  解答：Authentication 是確認身分，Authorization 是確認權限。
- 問題：前端把按鈕隱藏是否等於安全？  
  解答：不等於，真正控制點在後端授權。
- 問題：API Key 應如何管理？  
  解答：最小權限、週期輪替、分環境隔離、異常存取告警。
- 問題：401 與 403 如何區分？  
  解答：401 多為未驗證/驗證失敗；403 為已驗證但無權限。
- 問題：憑證外洩第一步做什麼？  
  解答：先止血（撤銷/輪替），再追查與補防護。

## 9.7 第 7 章提問與教師解答

### 題目 A
- 問題：SSE 斷線但 API 正常。
- 教師解答：優先檢查 SSE 端點可達性、Redis pub/sub 通道、前端重連與 fallback 邏輯。若 polling 正常，屬部分退化而非全站故障。

### 題目 B
- 問題：資料庫可用但畫面無更新。
- 教師解答：先驗證資料是否進到前端 store，再檢查 mapping/filter 條件與欄位命名是否變動，最後查快取與重繪邏輯。

### 題目 C
- 問題：告警資料有寫入但前端摘要卡空白。
- 教師解答：多半是查詢條件或欄位映射不匹配，也可能是排序/時間窗條件錯誤。先比對 API 回傳與 UI 期待 schema。

## 9.8 第 8 章提問與教師解答

### 提問 1
- 問題：為什麼專題要先做需求卡與不做清單？
- 教師解答：避免範圍膨脹。90 分鐘專題的關鍵是交付最小可運作版本（MVP），不是追求功能完備。

### 提問 2
- 問題：Demo 時應如何結構化說明？
- 教師解答：使用「需求 -> 做法 -> 結果 -> 風險」四段式，能同時展現工程能力與溝通能力。

### 提問 3
- 問題：評分時最看重什麼？
- 教師解答：先看功能正確性，再看穩定性與可讀性，最後看設計取捨說明。能清楚說明 why 比堆功能更重要。

## 9.9 手冊原有 FAQ 對照教師解答（補強版）

### Q1：新人該先學前端還是後端？

## 十、全檔一致格式版（問題 / 教師解答 / 追問）

> 使用方式：本版可直接作為講師口袋卡或助教對答稿。每題都採相同欄位，便於快速檢索與現場帶討論。

## 10.1 第 1 章：專案全貌與架構地圖

### 題目 1
- 問題：你認為資料中心監控系統最怕什麼？
- 教師解答：最怕「看不到、看太慢、看不準」。對應為可觀測性、低延遲、資料可信度。
- 追問：若只能選 1 件優先改善，你會選哪個，為什麼？

### 題目 2
- 問題：主路徑都正常時，為什麼還要設計降級路徑？
- 教師解答：主路徑保效能，降級路徑保可用。生產環境一定會有依賴故障，沒有降級就沒有韌性。
- 追問：哪個故障最適合先做降級演練？

### 題目 3
- 問題：請說出一筆資料從進站到畫面顯示的 5 個節點。
- 教師解答：接收 API -> 分流（Kafka/fallback）-> 處理（規則/標準化）-> 儲存（Influx/Mongo）-> 推送（SSE）-> 前端顯示。
- 追問：若推送失敗，哪個節點最先補救？

## 10.2 第 2 章：環境建置與啟動流程

### 題目 1
- 問題：一鍵啟動與手動啟動的教學價值差異是什麼？
- 教師解答：一鍵啟動重效率與一致性；手動啟動重理解與除錯。新人兩者都要會，才有維運能力。
- 追問：你會在什麼情境優先改用手動啟動？

### 題目 2
- 問題：環境檢查的推薦順序是什麼？
- 教師解答：版本與執行環境（Node/Python/Docker）-> 容器與埠 -> `.env` 設定 -> 應用 log。
- 追問：若時間只夠 3 分鐘，你先做哪兩項？

### 題目 3
- 問題：啟動失敗時第一步該做什麼？
- 教師解答：先縮小範圍，定義是「哪一層失敗」，再看該層 log。不要同時改多處。
- 追問：你如何避免「越修越亂」？

## 10.3 第 3 章：後端主流程與服務分層

### 題目 1
- 問題：為什麼 Router 要薄、Service 要厚？
- 教師解答：Router 專注協議與驗證，Service 集中業務規則，能降低耦合、提升可測與可維護性。
- 追問：若把業務寫在 Router，最先壞的是哪一件事？

### 題目 2
- 問題：新增遙測欄位預處理應放在哪？
- 教師解答：放在 service 或標準化流程，避免散落在路由邏輯，並維持向下相容。
- 追問：如何驗證舊資料不被破壞？

### 題目 3
- 問題：引入非同步工作佇列的最小侵入切點在哪？
- 教師解答：從 Service 層改造為事件投遞，維持路由契約不變，先保對外 API 穩定。
- 追問：你會先改哪一段耗時流程？

## 10.4 第 4 章：資料層與中介服務

### 題目 1
- 問題：為何 metrics 適合 Influx，不適合直接放 Mongo？
- 教師解答：metrics 多為時序查詢與聚合，Influx 對時間序列最佳化更完整，查詢效能與成本更佳。
- 追問：什麼情境你仍可能把部分 metrics 寫入 Mongo？

### 題目 2
- 問題：Redis 在本專案最關鍵的責任是什麼？
- 教師解答：即時事件分發（pub/sub）與 SSE 推送協作，不是長期主儲存。
- 追問：Redis 延遲時，前端該呈現什麼狀態？

### 題目 3
- 問題：怎麼判定是「降級可用」而非「完全不可用」？
- 教師解答：核心資料仍可接收/處理/落地，且前端可透過替代路徑顯示，即屬降級可用。
- 追問：你會設哪個指標來監控降級是否過久？

## 10.5 第 5 章：前端即時流與狀態管理

### 題目 1
- 問題：為何 fallback 不是錯誤狀態？
- 教師解答：fallback 是韌性策略，代表系統降級運作，不代表業務中斷。
- 追問：UI 上應如何提示，才不會讓使用者誤解成故障？

### 題目 2
- 問題：為什麼不把所有 state 都放全域？
- 教師解答：會增加耦合與重繪成本，降低可維護性。應依使用範圍切分 server state 與 UI state。
- 追問：你會如何判斷一個 state 應該全域化？

### 題目 3
- 問題：前端即時功能最常見的不穩定原因是什麼？
- 教師解答：`useEffect` 依賴錯誤導致重複連線，以及 cleanup 不完整造成 listener 殘留。
- 追問：你會加什麼觀測訊號來抓這類問題？

## 10.6 第 6 章：認證授權與安全治理

### 題目 1
- 問題：viewer 呼叫 control API，應回什麼？
- 教師解答：拒絕存取（通常 403），並保留可稽核資訊。
- 追問：401 與 403 的邊界如何向新人解釋？

### 題目 2
- 問題：token 過期時前端應如何處理？
- 教師解答：導向重新登入或 refresh 流程，保留必要上下文並避免無限重試。
- 追問：若 refresh 也失敗，你的 UX 回退策略是什麼？

### 題目 3
- 問題：若帳密誤上傳 repo，第一時間怎麼做？
- 教師解答：先止血（撤銷/輪替），再追查影響與補流程（secret scan、審查機制）。
- 追問：你會如何對團隊做事後教育？

## 10.7 第 7 章：除錯、監控與維運 SOP

### 題目 1
- 問題：SSE 斷線但 API 正常，先查什麼？
- 教師解答：先查 SSE 端點、Redis pub/sub、前端重連與 fallback 狀態。
- 追問：在什麼條件下你判定這是 P1 事件？

### 題目 2
- 問題：資料庫可用但畫面不更新，先查什麼？
- 教師解答：先查資料是否進前端 store，再查 mapping/filter 與欄位一致性。
- 追問：你如何快速確認是前端問題還是 API 問題？

### 題目 3
- 問題：告警有寫入但摘要卡空白，常見根因？
- 教師解答：查詢條件錯、欄位映射不匹配、時間窗或排序條件異常。
- 追問：你會先改哪個地方做最小驗證？

## 10.8 第 8 章：小專題實戰與驗收

### 題目 1
- 問題：為什麼要先寫需求卡與不做清單？

### 題目 3

## 第 4 章（新手版）：資料到底放哪裡

### 你只要記住
1. 不同資料放不同地方，是為了查得快、用得對。

### 教學活動（簡化）

## 第 5 章（新手版）：前端為什麼會即時更新

### 你只要記住

## 第 6 章（新手版）：權限與安全

### 你只要記住
1. 不是每個人都能做所有操作。
2. 前端看不到按鈕，不等於真的沒權限問題。
3. 真正權限檢查在後端。

### 教學活動（簡化）
- 講師示範不同角色登入，顯示不同可操作項目。

### 新手提問
- 問題：為什麼後端也要再檢查權限？

## 第 7 章（新手版）：除錯從哪開始

### 你只要記住

2. 一次只改一件事。
3. 要留紀錄，讓別人能重現。

### 新手 SOP（4 步就好）

4. 修完再驗證一次。

### 新手提問
- 問題：遇到 bug 為什麼不能一次改很多地方？

## 第 8 章（新手版）：做一個小功能

### 你只要記住
1. 先做最小可用版本（MVP）。
2. 功能要能 demo、能解釋。
3. 做完要說得出風險和下一步。

### 建議小題目（新手友善）
- 前端：新增一個狀態標籤（正常/警告）。
- 後端：新增一個簡單查詢 API（回傳固定格式）。
- 全端：新增一個按鈕，按下去能看到成功或失敗提示。

### 新手提問
- 問題：為什麼先做 MVP，不一次做滿？  
  教師解答：先做出可用版本才能早點驗證方向，避免做很多卻做錯方向。

## 11.4 新手驗收標準（簡化版）

符合以下 4 項即通過：

1. 可以把系統成功啟動。
2. 可以說出資料大致流程。
3. 可以完成一個小改動。
4. 可以寫出簡單問題排查紀錄。
