# 3D 機房設備通訊與實時監控技術說明文件

本文件說明 3D 數位孿生系統如何與監控資料流結合，並已依目前專案現況修正描述。

## 1. 系統架構概覽

監控系統目前採用 **Agent / Device -> Backend -> SSE -> 3D Frontend** 的資料驅動架構：

1. **數據採集層**：實體設備、模擬資料或 Agent 上送資料
2. **核心處理層**：FastAPI + Kafka + 資料處理
3. **即時廣播層**：以 `SSE` 為主，必要時由 Polling 保底
4. **視覺呈現層**：前端 `Next.js + React Three Fiber`

---

## 2. 各類設備通訊機制

### A. 伺服器與交換機 (Server & Switch)
-   **現況主幹**：以 Agent / 模擬資料 / HTTP ingest 為主。
-   **可整合方向**：SNMP、Redfish、gNMI 等。
-   **監控指標**：CPU 使用率、核心溫度、網路流量 (Gbps)、進口/出口溫差。
-   **監控策略**：目前由 `client_agent.py`、模擬資料與後端主資料流驅動，其他協定屬整合方向。

### B. CDU 液冷分配單元 (Liquid Cooling Unit)
-   **現況**：資料模型與視覺監控已納入專案。
-   **可整合方向**：`Modbus TCP/IP`、其他工控協定或設備 API。
-   **監控指標**：一次側/二次側供回水溫度、水泵轉速 (RPM)、流量 (LPM)、系統壓力 (bar)。
-   **監控策略**：CDU 通常配備 PLC 控制器，監控主機透過讀取特定暫存器 (Registers) 獲取精密流體數據。

### C. 雙相浸沒式冷卻機櫃 (Immersion Cooling)
-   **現況**：已納入 3D 與展示敘事。
-   **可整合方向**：`Modbus TCP/IP`、`MQTT` 或客製設備介面。
-   **監控指標**：冷卻液高度 (Coolant Level)、儲槽內部壓力、氟化液溫度、熱交換器效率。
-   **監控策略**：由於雙相浸沒涉及相變過度壓力，系統需具備「高頻次採樣」能力（每秒 2-5 次），確保壓力容器安全。

## 3. 即時監控技術核心

### SSE (Server-Sent Events) 廣播機制
本系統的正式即時更新通道為 SSE：
-   **單向持續流**：服務器主動向前端「推播」數據。
-   **輕量化**：相比 WebSocket，SSE 在處理大量的設備監控數據時更為穩定且開銷較小。
-   **自動重連**：若網路波動中斷，前端會重試並可退回 Polling。

### 數據映射與 3D 渲染 (Data Mapping)
1.  **ID 匹配**：監控數據帶有 `server_id`，前端接收時會自動匹配 3D 場景中具有相同 ID 的物件。
2.  **動態視覺化**：
    *   **Server**：根據 CPU 負載改變伺服器燈號顏色（藍→黃→紅）。
    *   **CDU**：根據溫度高低改變水流特效的移動速度與顏色深度。
    *   **浸沒槽**：根據液位感測器數值，實時調整 3D 模型中的液面高度 (Liquid Height)。
