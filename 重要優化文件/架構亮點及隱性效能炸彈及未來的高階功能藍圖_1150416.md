## 1. 目前架構亮點

### 1.1 前端

- 實際框架為 `Next.js 16.2.0`
- 採 `SSE 優先、Polling 保底` 的即時更新策略
- 使用 `Zustand` 管理 3D 場景與前端狀態
- 已引入 `idb-keyval`，可作為大型持久化資料的方向

### 1.2 後端

- 核心框架為 `FastAPI`
- 以 `AppContainer` 統一組裝服務與背景工作
- 主資料流為 `HTTP /ingest -> Kafka -> Backend processing -> InfluxDB / MongoDB -> SSE`
- `SSEManager` 已具備即時廣播能力，並支援 Redis 背板方向

### 1.3 儲存與中介

- `Kafka`：事件匯流排
- `InfluxDB`：時序資料
- `MongoDB`：告警、使用者、維運資料
- `Redis`：SSE Pub/Sub 輔助

## 2. 已知高風險區域

### 2.1 多實例 SSE 一致性

- 風險：若未妥善使用 Redis 背板，多 worker / 多 instance 可能出現 SSE 孤島
- 建議：正式環境以 Redis Pub/Sub 或等效事件背板統一廣播

### 2.2 前端大型場景持久化

- 風險：大量 3D 配置若完全使用同步型 `localStorage`，可能造成容量與效能問題
- 建議：大型結構資料以 IndexedDB 或後端持久化為主

### 2.3 高頻寫入負載

- 風險：高頻單筆寫入 InfluxDB 會增加 I/O 負擔
- 建議：評估 batching、buffer 與更完整的寫入保護機制

### 2.4 文件與實作差異

- 風險：舊文件仍可能出現 Next 15、WebSocket、舊埠號、完全落地的 Redfish/Modbus 等描述
- 建議：統一以本輪更新後的正式文件為準
