# DataCenter 開發與除錯日誌 (Debug Log)
**日期：** 2026-03-23

今日針對系統無法順利啟動、前端無資料顯示、以及連線逾時等綜合性問題進行了深度除錯。許多問題是源自於 Windows 作業系統的獨有特性或設定，以下詳細記錄問題成因、解決方案，以及未來移機至別台電腦的預防指南。

---

## 1. 系統組件（Docker / Kafka / InfluxDB）

### 🛑 狀況 A：Kafka 預設 9092 Port 衝突
- **問題描述：** `docker-compose up` 啟動時失敗，出現 `Ports are not available... An attempt was made to access a socket...`
- **根本原因：** Windows 系統會啟動 Hyper-V 與 NAT 服務以供應 WSL2 與 Docker 運行，這些服務會獨佔且隨機保留某些通訊埠（剛好踩到原先配置的 9092 與 9093）。
- **解決方案：** 於 `docker-compose.yml` 中將對外映射 Port 改為較高頻段的安全區間 `29092`。
- **未來是否還會發生：** **【絕對不會】** ✅。配置檔已更新入 Git，未來在任何電腦啟動都會自動避開這個衝突。

### 🛑 狀況 B：Kafka 內部/外部網路解析失敗 (`KafkaTimeoutError`)
- **問題描述：** 後端 Python 不斷拋出 `Failed to update metadata after 60.0 secs.`，無法發送也無法消費數據。
- **根本原因：** Kafka Docker 容器需要知道自己對「內部網路」與「外部網路（宿主機）」分別該廣播哪條位址。原本設定只有一條，導致 Kafka Controller 連線混亂。
- **解決方案：** 在 YAML 檔中明確宣告了雙向 Listener (`INSIDE://0.0.0.0:9092`, `OUTSIDE://0.0.0.0:29092`) 來分離內網與外網請求。
- **未來是否還會發生：** **【不會】** ✅。此雙向設定極度穩定，跨機器佈署也能自動生效。

### 🛑 狀況 C：InfluxDB 寫入權限拒絕 (`401 Unauthorized`)
- **問題描述：** 後端讀到 Kafka 資料要寫入 InfluxDB，但回應未授權，導致歷史資料無法儲存。
- **根本原因：** `influxdb:2.0` 在初始化階段，只有建立帳號密碼，卻漏給了寫入專用的 API Admin Token。
- **解決方案：** 在 YAML 中補上 `DOCKER_INFLUXDB_INIT_ADMIN_TOKEN = adminpassword`。
- **未來是否還會發生：** **【不會】** ✅。下次啟動全新的 Container 日誌時就會自動幫您設定好。

## 2. 後端與通訊層（Python / Uvicorn）

### 🛑 狀況 D：Windows 上的幽靈處理程序 (Ghost Processes)
- **問題描述：** 系統顯示「狀態一切正常、Producer 與 Consumer 皆連線」，前端卻只收到毫無內容的 `[]` 空陣列。
- **根本原因：** 你每次點選 `start.bat` 執行之後又在 Windows 直接按 `X` 關閉黑窗，導致底層的 Python 工作程序 (Uvicorn Worker) 實質上並沒有被砍死（殘留了多達 6 個舊行程）。這些已經壞掉的幽靈程式因為 Socket 多工特性，在背景把新的 API 呼叫給「攔截」過去了。
- **解決方案：** 在 `cmd` 利用 `taskkill /F /IM python.exe` 將所有死機的程式強制清道夫。
- **未來是否還會發生：** **【可能會】** ⚠️。這受限於開發者個人的操作習慣。未來在「關閉、重啟」專案前，建議養成習慣先進入那個黑色終端機畫面按下 `Ctrl + C`，確認程式優雅關閉後再打叉。

### 🛑 狀況 E：Python Kafka 套件的 Socket 崩潰 (`ValueError: Invalid file descriptor: -1`)
- **問題描述：** 後端的佇列消費執行緒 (Thread) 因為這行報錯直接閃退。
- **根本原因：** 這是 `kafka-python` 函式庫在 Windows `selectors.py` 處理網路異步讀寫的一個知名底層 Bug（Linux / Mac 上不會發生）。只要稍有延遲或網路斷層就會踩雷。
- **解決方案：** 取代直接修復套件底層，我將它包進一個具有「自我治癒能力 (Auto-healing)」的 `try-except-retry` 無限迴圈中。
- **未來是否還會發生：** 底層依然會發生，**【但你再也不會受到影響】** ✅。現在只要它一崩潰，系統 1 秒內就會將舊連線拋棄並滿血復活續跑。

## 3. 前端與視覺層（Next.js / 3D）

### 🛑 狀況 F：數位分身 (3D Digital Twin) 的顏色與狀態不同步
- **問題描述：** 首頁已經有資料且 5 秒更新一次，但切換到 3D 頁面時，伺服器模型都不發光也不變色。
- **根本原因：** Next.js 的預設 `fetch()` 機制會去偷快取舊的 API 回傳結果。
- **解決方案：** 在針對這類需要即時活度資料的地方，主動標記 `{ cache: "no-store" }`，並加上每 5 秒的定期輪詢。
- **未來是否還會發生：** **【不會】** ✅。程式碼已強制同步更新機制。

### 🛑 狀況 G：家裡新增的機台，在新電腦開專案變不見了
- **問題描述：** 昨天回家拉出的 3D 機房，使用 Git 抓下最新版程式碼後未顯示。
- **根本原因：** 該拓樸資料狀態是由 `zustand` 管理並保存在瀏覽器的 本地儲存空間 (`localStorage`) 裡。這代表「你在哪台螢幕上拉設配置，該配置就保留在那台機器的 Google Chrome 裡」。
- **解決方案：** 
  - 若在開發初期：可以在網頁 UI 上先拉出滿意的樣貌，點選「Download (匯出)」存成 `.json` 之後，再到公司電腦「Upload (匯入)」。
  - 若確定這就是預設模板：可以直接將這份 `dcim_layout.json` 打開，把裡面的內容寫死貼進前端程式碼中（取代由 UUID 隨機生成的模板），作為未來任何人開啟專案的預設場景。
- **未來是否還會發生：** 這是正常行為，**不算是錯誤**。未來只要有不同設備的操作就會面臨相同情境。
